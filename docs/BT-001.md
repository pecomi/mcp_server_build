# BT-001 — Backend function-level authz on restricted store rows

## [Defense ID]

BT-001 (mcp_server_build track, first defense).

## [Targets]

- **RT-002 Stage 1** — primary target. Closes the **classical leg** (B4 IDOR) of RT-002, independent of MCP-layer integrity.
- Future single-server IDOR-class RTs against `/stores/{id}` — also closed.

## [Mechanism]

**Where**: `mock-backend` `PublicStoreController.getStore` (server middleware, classical layer).

**What**: when env `BT_BACKEND_AUTHZ_ENABLED=true`, the controller checks the requested row's `restricted` flag *after* repository lookup. If `restricted=true`, returns `403 RESTRICTED` with a directive to use the bearer-gated `/secure/stores/{id}` endpoint instead. Non-restricted rows (STORE-001/002/003) are unaffected.

Code: [`mock-backend/src/main/java/mock_backend/api/PublicStoreController.java`](../mock-backend/src/main/java/mock_backend/api/PublicStoreController.java).

This is BT-CANDIDATES.md option C-2 — "단순 `restricted` 플래그 gating", the cheapest of the three options.

## [Coverage]

- **Full**: RT-002 Stage 1 — fs (description poisoning) + H (LLM follows directive) still active, but the resulting `tools/call getStoreDetail{store_id:"STORE-INTERNAL-001"}` reaches backend, which returns 403 instead of leaking. `finalText` does not contain restricted fields. *Attack neutralized at the classical layer.*
- **Partial**: RT-003 — unaffected by BT-001 because RT-003 targets fs-server (`read_file`), not mock-backend. BT-001's scope is `/stores/{id}` only.
- **None**: other future RTs (path traversal on fs-server, FSP, etc.) — orthogonal sinks.

**Important — orthogonality to S and H flags**:
- S (description poisoned) flag stays ON; description is still poisoned in tool catalog. **Scanner BT-A would catch this independently.**
- H (host LLM follows directive) stays ON; LLM still attempts the malicious arg. **BT-B (host sanitization) would catch independently.**
- BT-001 just kills the *result*. caller sees `Tool getStoreDetail error: <403 RESTRICTED ...>` in `finalText`.

This is cross-layer defense-in-depth's "honest cost" demonstration: each BT closes a different layer; combinations are tighter.

## [Cost]

- **Code**: 1 if/return + 1 `@Value` injection (~5 lines).
- **Latency**: negligible — single boolean check after Map lookup.
- **False positive**: 0% — `restricted` is a binary domain flag, no heuristics. Public rows pass through unchanged.
- **False negative on intent**: if a row is misclassified (e.g., should-be-restricted row has `restricted=false`), BT-001 doesn't help. Classification quality is upstream (data steward responsibility).
- **Dev burden**: trivial. Single env toggle for rollout/rollback.

## [Bypass surface]

- **Different sink, same data class**: attacker pivots to `/secure/stores/{id}` directly with Bearer token (if token leaks elsewhere). Out of BT-001 scope — that's BT for the secure endpoint.
- **Different sink, different data class**: BT-001 is store-specific. If another endpoint (e.g., hypothetical `/users/{id}` or fs-server `/data/restricted/...`) becomes a sink, BT-001 doesn't help. Each sink needs its own gate (or a generic per-row authz layer at a higher level).
- **Description-poisoning chain to a different tool**: if attacker poisons `getStoreList` description with a different exfil pattern, BT-001's row-level gate still applies (list endpoint filters restricted anyway), but a creative attacker might find a tool that bypasses both.
- **Toggle bypass**: BT_BACKEND_AUTHZ_ENABLED set to `false` (insider / config compromise) → BT off. Operationally, this env should be locked at infra level, not at app config.

## Verification

Automated 4-case harness:

```bash
bash scripts/verify-bt-001.sh
```

- Case 1: BT off, `/stores/STORE-INTERNAL-001` → 200 (baseline IDOR sink alive — confirms BT was truly off).
- Case 2: BT on, `/stores/STORE-INTERNAL-001` → 403 RESTRICTED.
- Case 3: BT on, `/stores/STORE-001` → 200 (no regression on public rows).
- Case 4: BT on + RT-002 attack (S=ON, H=ON) → caller's `finalText` does NOT contain `VIP 의전용`. **Attack blocked end-to-end.**

DoD: 4/4 PASS, `RESULT: BT-001 DoD MET`, exit 0. Script restores compose state (BT off, honest descriptions, mock host) on exit.

## Activation

Production-style: set `BT_BACKEND_AUTHZ_ENABLED=true` in compose env, then `docker compose up -d --force-recreate mock-backend`. RT-002 reproduction continues to demonstrate the attack vehicle, but the result chain breaks at the backend.

Default ships as `false` so that RT-002 demo + Stage 1 verification remain reproducible without explicit toggle.
